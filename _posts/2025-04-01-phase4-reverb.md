> *"Реверб — це просто затримки. Я думав, що це буде легко."*
> *(Спойлер: не було.)*

**Ціль:** цифровий пружинний реверб. A0 (гучність), A1 (wet/dry). Три delay lines, decay, saturation.

---

## Теорія: що таке реверб

- **Delay** = одне чітке відлуння — чуєш окремий повтор
- **Reverb** = десятки дрібних відлунь що зливаються — чуєш **простір**

Пружинний реверб: затримки нерівномірні, характерний "boing" — фізично це металева пружина в старих гітарних комбо Fender. Ми імітуємо це цифрово.

```
audio_in ──┬── × dry_gain ───────────────────────────┐
           ├── delay 1100 семплів × decay 0.75 → +   │
           ├── delay 2250 семплів × decay 0.60 → +   │
           └── delay 3900 семплів × decay 0.45 → +   │
                                         ÷ 2          │
                                       × wet_gain     │
                                              └── + ──┘
                                                  │
                                              audio_out
```

---

## Архітектура: delay_line.v

Circular buffer у Block RAM (M9K). Ідея: стрічка конвеєра довжиною 4096 комірок. Кожен такт кладеш новий семпл на один кінець і читаєш старий з іншого.

```verilog
reg [ADDR_BITS-1:0] wr_ptr = 0;
wire [ADDR_BITS-1:0] rd_ptr = wr_ptr - delay_samples;

always @(posedge clk) begin
    if (we) begin
        mem[wr_ptr] <= audio_in;
        wr_ptr      <= wr_ptr + 1'd1;
    end
    audio_out <= mem[rd_ptr];
end
```

**Магія з переповненням:** `wr_ptr` — 12-бітне число (0..4095). Від'ємних значень не буває — число просто обертається:

```
wr_ptr = 10,  delay = 50
rd_ptr = 10 - 50 = -40  →  в 12-бітному числі: 4096 - 40 = 4056  ✅
```

Ніяких `if (ptr >= DEPTH) ptr = 0` — арифметика переповнення робить це сама.

## Fixed-point арифметика (Q1.15)

FPGA не має апаратної підтримки `float`. **Fixed-point** — домовленість між програмістом і залізом:

> Я зберігаю ціле число, але ми обидва знаємо що треба ділити на 32768.

Пишеш `24576`, маєш на увазі `0.75`. **fixed-point = ціла арифметика + домовленість про масштаб**.

```
delay 23ms → decay 0.75 × 32768 = 24576  (найгучніше ехо)
delay 47ms → decay 0.60 × 32768 = 19661
delay 81ms → decay 0.45 × 32768 = 14746  (найтихіше ехо)
```

```verilog
wire signed [31:0] product = $signed(sample_in) * $signed({1'b0, DECAY[14:0]});
wire signed [15:0] result  = product[30:15];  // беремо [30:15] — це >> 15
```

## Проблема 1: reverb_core — нуль на виході

Написав `reverb_core.v`. Залив. Звук є. Реверб — **нуль**.

**Діагностика:** вирізав `reverb_core`, підключив один `delay_line` напряму — чітке ехо через 50ms. `delay_line` живий. Проблема в `reverb_core`.

> 🔬 Вирізай блоки і тестуй мінімальний шлях — заощаджує дні гадань.

---

## Проблема 2: ADS1115 multi-channel — continuous mode не перемикає MUX

ADS1115 має 4 аналогових входи, але один АЦП. Перемикання між каналами через внутрішній **MUX**.

В **continuous mode** чіп ігнорує зміну MUX поки конвертує. **Рішення — single-shot mode:**

```verilog
wire [7:0] cfg_msb = ch ? 8'hD3 : 8'hC3;
// 0xC3 = OS=1, MUX=AIN0, single-shot
// 0xD3 = OS=1, MUX=AIN1, single-shot
```

Цикл: конфіг → чекаємо 15ms → читаємо → перемикаємо канал.

---

## Проблема 3: Channel bleeding — 10ms замало

**Конверсія** — час що ADS1115 витрачає на вимірювання. При 128 SPS: 1/128 ≈ 7.8ms, з допуском ±10% → до **8.6ms**.

10 - 8.6 = 1.4ms. З I²C overhead іноді читали до завершення — сміття з попереднього каналу.

```verilog
// Було:  15'd3999  → 10ms
// Стало: 15'd5999  → 15ms
```

> ⏱ "З запасом" в hardware — це не +20%. Це ×2 мінімум.

---

## Проблема 4: Torn reads — 90мкс вікно з брудними даними

ADS1115 повертає результат двома байтами (MSB і LSB) з ~90мкс між ними. Якщо читати `volume_raw` між ними — половина старого + половина нового.

Java-аналогія: `long` на 32-бітній JVM без `synchronized`.

**Рішення — атомарний запис через буфер:**

```verilog
// Збираємо в буфер:
S_R_MSB  → adc_data[15:8] <= rxd;
S_R_LSB  → adc_data[7:0]  <= rxd;
// Пишемо одним присвоєнням коли обидва байти готові:
S_R_STOP → volume_raw <= adc_data;
```

> 🔒 Multi-byte регістри на hardware = race condition в multi-threaded коді.

---

## Проблема 5: wet_gain не дає ефекту

dry=100% маскував wet. Вухо звикає до "основного" звуку і не чує добавку. Рішення — **crossfade**:

```verilog
wire [15:0] wet_gain = wet_raw[15] ? 16'h0000 : {1'b0, wet_raw[14:0]};
wire [15:0] dry_gain = 16'h7FFF - wet_gain;
```

A1=max → dry≈0.2, wet=0.8. Покрутив A1. **Почув хвіст. Гулкий, з decay, з простором.**
